# Handover — Phase 0.2: Health endpoint, Serilog, OpenTelemetry

**Дата:** 2026-05-14
**Roadmap:** [Фаза 0 — Скелет](../10-roadmap.md), задачи «`Sigil.Api` отвечает `200 OK` на `/health`» и «Serilog → stdout, OpenTelemetry заглушка».

## Что сделано

### Sigil.Api

- **Serilog** настроен:
  - Bootstrap-logger в `Console` + `CompactJsonFormatter` (одна строка JSON на запись) до окончательной конфигурации хоста.
  - После `Host.UseSerilog(...)` — конфиг подхватывается из `appsettings*.json`, сервисы (например, `IConfiguration`) доступны как обогатители.
  - Обогащение: `MachineName`, `EnvironmentName`, статический property `service=sigil-api`, `FromLogContext` для scoped-полей.
  - `UseSerilogRequestLogging()` пишет одну сводную строку на HTTP-запрос (метод, путь, статус, длительность) — заметно меньше шума, чем дефолтное логирование ASP.NET Core.
  - Глобальный `try/catch + Log.Fatal + CloseAndFlushAsync` в `finally` гарантирует, что unhandled exception попадает в лог даже при crash'е.

- **OpenTelemetry** настроен как placeholder:
  - Resource объявлен с `service.name=sigil-api`, `service.version` = `Assembly.Version`.
  - Traces: `AspNetCoreInstrumentation` + `HttpClientInstrumentation`.
  - Metrics: те же + `RuntimeInstrumentation`. Зарегистрирован Meter prefix `Sigil.*` для собственных счётчиков, которые мы добавим позже.
  - **OTLP exporter** включается **только если** `Otel:Endpoint` задан в конфиге. По умолчанию в `appsettings.json` — пустая строка → экспортёр не подключён, но collection работает (метрики и spans можно посмотреть, например, через диагностический dump). Это и есть «заглушка» из roadmap.

- **Health endpoints:**
  - `GET /health` — кастомный JSON: `{ status: "ok", service: "sigil-api" }`. Удобно для smoke-проверок и для дашборда.
  - `GET /health/live` — стандартный ASP.NET Core HealthCheck (200 пока всегда; в Phase 0.3 добавим DB-check).
  - `GET /health/ready` — то же. Разделение `live/ready` нужно для kubelet/balancer: `live=200` пока процесс жив, `ready=200` пока зависимости доступны.

### Конфигурация

- `appsettings.json` теперь содержит секцию `Serilog:MinimumLevel` и `Otel:Endpoint`. Старая секция `Logging` удалена — она бы перекрывалась Serilog'ом и только запутывала.
- `appsettings.Development.json` понижает порог логирования до `Debug` для разработки.

## Принятые решения по ходу

1. **`Microsoft.Extensions.Diagnostics.HealthChecks` отдельным пакетом не подтягиваем** — он уже входит в framework reference `Microsoft.AspNetCore.App`. (Попытка `dotnet add package` свалилась с `Value cannot be null (Parameter 'path1')` — это известная странность NuGet для метапакетов, входящих в shared framework. Просто не нужен.)

2. **OTLP-эндпоинт по умолчанию пуст.** В Phase 7 (observability) поднимем Tempo/Prometheus и завязываемся через env-var `Otel__Endpoint=http://otel-collector:4317`. Сейчас — заглушка, как и просил roadmap.

3. **Bootstrap-logger обязателен** — иначе любой crash до `Host.Build()` (например, при ошибке парсинга конфига) уйдёт в `Console.WriteLine` без структуры. Compact JSON оставляет crash логируемым.

4. **`UseSerilogRequestLogging` поверх стандартного логирования** — стандартный middleware ASP.NET Core пишет три строки на запрос (Hosting/Started, Routing/Matched, Hosting/Finished). Serilog-version даёт одну `HTTP {Method} {Path} responded {StatusCode} in {Elapsed:0.0000} ms`. Это снижает объём логов в проде в 3-4 раза.
