# 00 — Обзор и цели

## Проблема

Разработчик продаёт софт, который разворачивается на серверах заказчиков (on-prem) и не всегда имеет стабильный интернет. Нужны:

1. Криптографически защищённые лицензии, которые невозможно подделать без приватного ключа разработчика.
2. Возможность работать оффлайн до 30 дней.
3. Гибкая конфигурация (feature flags, лимиты) — внутри самой лицензии, чтобы клиентский код не ходил на сервер для каждой проверки фичи.
4. Управление иерархией клиентов (агентств с дочерними структурами).
5. Биллинг (планы, подписки, инвойсы).
6. Веб-панель для администраторов и клиентов.
7. **Self-hosted deployment** (PostgreSQL, MinIO, Vault, observability — всё на своих серверах; единственное внешнее — Cloudflare как edge).

## Цели (что входит)

- **License Service** — выпуск, продление, отзыв, верификация лицензий.
- **Tenant Hierarchy** — дерево компаний произвольной глубины.
- **License Templates** — переиспользуемые шаблоны конфигурации лицензии.
- **Billing** — планы, подписки, инвойсы, метеринг (минимально — флаги и лимиты).
- **Web Panel** — React-приложение для админов и клиентов.
- **Client SDK** (.NET) — библиотека для валидации лицензий внутри продуктов заказчика; единый эндпоинт `sigil.hizmatrason.tj` для refresh/heartbeat.
- **Webhooks** — события для интеграций tenant'ов.
- **Audit Log** — кто что когда изменил.

## Что НЕ входит в v1 (non-goals)

- Авто-выдача поддоменов на лицензию (отменено — единый API-хост `sigil.hizmatrason.tj`).
- Интеграция с SaaS-провайдерами платежей (Stripe и т.п.) — нарушает self-hosted-политику; используем ручное/банковское признание оплат.
- Cloud KMS (AWS / Azure / Yandex) — приватные ключи только локально (encrypted file) или в self-hosted Vault.
- Сложные финансовые операции (мультивалютность, налоги по странам).
- Глубокая аналитика использования (telemetry pipeline) — заложим хуки, но дашборды позже.
- Marketplace / партнёрские программы.
- Поддержка нескольких языков UI (только RU/EN на старте).
- Mobile SDK (Android/iOS) — только .NET в v1.

## Глоссарий

| Термин | Значение |
|--------|----------|
| **Tenant / Company** | Узел в дереве владельцев. Корневой = прямой клиент Sigil; потомки = его суб-клиенты. |
| **License Template** | Шаблон, описывающий продукт, схему конфигурации, базовый домен, тарификацию. |
| **License** | Экземпляр шаблона, выданный конкретной компании, с заполненной конфигурацией. |
| **License Token** | Подписанный сервером пакет, который клиентский SDK валидирует. |
| **Activation** | Привязка лицензии к конкретному инстансу (HW fingerprint / host). |
| **Heartbeat** | Периодический check-in клиента на сервер (по умолчанию раз в 24ч). |
| **Grace Period** | Период (30 дней), в течение которого клиент работает без связи с сервером. |
| **Operator** | Сотрудник компании-владельца Sigil (super-admin). |
| **Tenant Admin** | Администратор компании-клиента. |
| **Product** | Конкретный софт, на который выдаётся лицензия (поле в шаблоне). |

## Целевые сценарии (user stories)

1. **Оператор Sigil** создаёт компанию-клиента, выдаёт ей админ-доступ и тарифный план.
2. **Tenant Admin** создаёт шаблон лицензии для своего продукта: задаёт JSON Schema конфигурации и оффлайн-период.
3. **Tenant Admin** создаёт **суб-компанию** (например, своего реселлера) и делегирует ей право выпускать лицензии только из определённого шаблона.
4. **Tenant Admin** выпускает лицензию для конечного клиента: заполняет конфиг → получает подписанный `.sigil` файл + публичный ключ для встраивания в продукт.
5. **Клиентский SDK** в продукте заказчика валидирует токен оффлайн, каждые 24 часа делает heartbeat в `sigil.hizmatrason.tj` для получения свежей версии (если конфиг был изменён в панели).
6. **Конечный клиент** теряет интернет — продукт продолжает работать 30 дней, затем переходит в degraded mode.
7. **Tenant Admin** отзывает лицензию — все клиентские инстансы при следующем heartbeat получают `revoked` и блокируются.

## Принципы дизайна

- **Оффлайн-first для клиентов.** Сервер недоступен ≠ продукт сломан.
- **Подписанные payload вместо запросов.** Конфиг живёт в самом токене, не в API.
- **Multi-tenant с самого начала.** Изоляция данных по tenant_id — обязательна.
- **Каждое действие в панели — аудируется.**
- **Никаких секретов в клиентских билдах.** Только публичный ключ.

## Ограничения и предположения

- Целевая нагрузка v1: до 1000 tenant'ов, до 100k активных лицензий, ~1M heartbeat'ов/день. Один инстанс PostgreSQL + один app-сервер справится.
- Лицензии rarely-write, often-verify: основная работа SDK — локальная проверка подписи.
- **Self-hosted-first.** Все компоненты можно поднять на одной VM с docker-compose. Cloudflare — единственный внешний сервис и только в роли edge.
